# EKS Pod Failure Investigation Runbook

## CRITICAL: Data Source Priority

**S3 collected data is the PRIMARY and PREFERRED source for investigation.**

### Why S3 Data First?
- S3 contains point-in-time snapshot captured at failure moment
- Pod may have been restarted, rescheduled, or deleted since failure
- Node logs (dmesg, kubelet, containerd) are NOT accessible via kubectl
- Previous container logs may be lost after restart
- S3 data is complete and immutable

### Investigation Order (MANDATORY)
1. **FIRST**: Read S3 collected data using the EXACT path provided in incident context
2. **SECOND**: Only use kubectl/EKS access if S3 data is insufficient or missing

---

## Step 1: Locate S3 Data Path

The incident context contains the **EXACT S3 path**. Use it directly.

### S3 Path Structure (Timestamp First)
```
s3://devops-agent-operator-archive/incidents/<timestamp>/<namespace>/<pod-name>/
```

Example:
```
s3://devops-agent-operator-archive/incidents/2025-02-09T12-00-00Z/test/oom-test/
```

### Timestamp Format
- Format: `YYYY-MM-DDTHH-MM-SSZ`
- Timezone: UTC
- Example: `2025-02-09T12-00-00Z` means February 9, 2025 at 12:00:00 UTC

### Finding the Correct Incident by Timestamp

If you need to search for the incident data:
1. The incident timestamp in the webhook payload indicates when the failure occurred
2. Look for the S3 path with timestamp **closest to and before** the incident time
3. Timestamps are sorted chronologically - recent incidents appear later alphabetically

Example: If incident occurred at `2025-02-09T12:05:00Z`:
- Look for paths starting with `2025-02-09T12-`
- Choose the one closest to but not after `12:05:00`

### IMPORTANT: Direct Access Method
- **DO NOT** use S3 ListObjects to search through all incidents
- **DO** use S3 GetObject with the exact path from incident context
- The full path is provided in the incident - use it directly

---

## Step 2: Read collected-data.json First

**Always start with `collected-data.json`** - this single file contains ALL diagnostic data.

Path: `<s3-base-path>/collected-data.json`

Example:
```
s3://devops-agent-operator-archive/incidents/2025-02-09T12-00-00Z/test/oom-test/collected-data.json
```

This file includes:
- Pod manifest and describe output
- Container logs (current and previous)
- Node logs (kubelet, containerd, dmesg)
- Failure information

---

## Step 3: Analyze by Failure Type

### For OOMKilled:
1. `failure-info.json` - Confirm failure type and exit code (137)
2. `node-logs/dmesg.log` - Search for "Out of memory", "oom-kill", "Killed process"
3. `pod-manifest.yaml` - Check resource limits and requests
4. `node-logs/mem-usage.txt` - Check node memory pressure

### For CrashLoopBackOff:
1. `logs/<container>-previous.log` - Logs before crash (CRITICAL - not available via kubectl after restart)
2. `logs/<container>.log` - Current container logs
3. `pod-describe.yaml` - Exit codes, restart count, events
4. `node-logs/containerd.log` - Container runtime errors

### For Error/Failed:
1. `failure-info.json` - Check exit code and reason
2. `logs/<container>.log` - Application error logs
3. `pod-describe.yaml` - Pod events and conditions

---

## Step 4: Infrastructure Analysis (S3 Only)

These logs are ONLY available in S3, NOT via kubectl:

| S3 File | Information | kubectl Equivalent |
|---------|-------------|-------------------|
| node-logs/dmesg.log | OOM killer, kernel errors | NOT AVAILABLE |
| node-logs/kubelet.log | Pod lifecycle, image pull | NOT AVAILABLE |
| node-logs/containerd.log | Container runtime | NOT AVAILABLE |
| node-logs/ipamd.log | VPC CNI, network issues | NOT AVAILABLE |
| node-logs/disk-usage.txt | Node disk status | NOT AVAILABLE |
| node-logs/mem-usage.txt | Node memory status | NOT AVAILABLE |
| logs/*-previous.log | Pre-crash logs | May be lost after restart |

---

## Step 5: When to Use kubectl/EKS Access

Only use kubectl or EKS access when:
- S3 data path is not provided in the incident
- S3 files are missing or corrupted
- You need to check CURRENT state (not failure-time state)
- You need to check related resources not in S3 (Services, ConfigMaps, Secrets)

**Remember**: kubectl shows CURRENT state, S3 shows FAILURE-TIME state. For root cause analysis, failure-time state is more valuable.

## S3 File Quick Reference

| File | Priority | Contains |
|------|----------|----------|
| collected-data.json | HIGHEST | All data in single JSON - START HERE |
| failure-info.json | HIGH | Failure type, container, exit code |
| logs/*-previous.log | HIGH | Pre-crash container logs |
| node-logs/dmesg.log | HIGH | OOM events, kernel errors |
| pod-describe.yaml | MEDIUM | Pod status, events, conditions |
| pod-manifest.yaml | MEDIUM | Full Pod spec, resource limits |
| node-logs/kubelet.log | MEDIUM | Pod lifecycle issues |
| logs/*.log | MEDIUM | Current container logs |

## Quick Start Checklist

1. [ ] Find S3 path in incident context (format: `s3://.../incidents/<timestamp>/<namespace>/<pod>/`)
2. [ ] Verify timestamp matches incident time (closest timestamp before incident)
3. [ ] Use S3 GetObject to read `collected-data.json` at that path
4. [ ] Check `failure-info.json` for failure type
5. [ ] Follow failure-type-specific analysis steps above
6. [ ] Only use kubectl if S3 data is missing or need current state
